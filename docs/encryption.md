# End-to-End Encryption

This document describes the end-to-end (E2E) encryption between this client and
the IPTranslator GPU translation service, as implemented in this repository.

## Threat model and design goal

The IPTranslator web app authenticates API keys, enforces quotas, and registers
consumption — but it must **never be able to read translation content**.
Payloads are encrypted on the client and decrypted only on the GPU worker
(which must see plaintext to translate). The web app forwards opaque messages
and reads only unencrypted envelope metadata (`consumed_units`, `object_id`,
languages, timing) to do its job.

Consequences:

- Any key the client *receives from the web app* is untrusted by default.
  Trust is established either by a key the client pins locally or by a
  signature the web app cannot forge (see *Key trust* below).
- All encryption is **post-quantum**: recorded traffic must stay confidential
  and key announcements must stay unforgeable even after a cryptographically
  relevant quantum computer exists.

## Message encryption

Implemented in [`TranslatorClientMessageHandler`](../src/IPTranslator.Client/Messaging/TranslatorClientMessageHandler.cs)
and [`XWingKem`](../src/IPTranslator.Client/Messaging/XWingKem.cs).

Per client session, one **X-Wing** encapsulation (draft-connolly-cfrg-xwing-kem:
ML-KEM-768 + X25519 combined with SHA3-256) against the service's 1216-byte
public key yields the session AES key plus the 1120-byte `encrypted_key` that is
sent with every message under a random `key_id`. Message bodies are encrypted
with **AES-256-GCM** (12-byte nonce in `entropy`, 16-byte tag appended to the
body). The GCM associated data binds the unencrypted envelope fields, so they
cannot be altered independently of the body:

```
requests:  UTF8("iptv3-req:v1:" + request_id + ":" + key_id)
responses: UTF8("iptv3-rsp:v1:" + request_id + ":" + key_id + ":" + request_type
                + ":" + consumed_units + ":" + base64(object_id))
```

Binding `consumed_units` and `object_id` into the response AAD means the billing
fields the web app reads in plaintext are also *verified end-to-end*: a relay
that tampered with them would fail GCM authentication on the client.

## Document blob encryption (`iptd-doc:v1`)

Implemented in [`DocumentBlobCipher`](../src/IPTranslator.Client/Messaging/DocumentBlobCipher.cs).

Documents of a batch job do not travel through the request envelope; they are
uploaded directly to blob storage (claim-check pattern) encrypted with
**AES-256-GCM** under a fresh per-job 256-bit blob key generated by the client.
Blob layout: `12-byte nonce || ciphertext || 16-byte tag`. The GCM AAD binds job
identity and blob slot, so a blob cannot be replayed in another slot or under
another job:

```
UTF8("iptd-doc:v1:" + job_id + ":" + slot)     slots: in, out, dict, tm0..tmN
```

The blob key travels only **inside the encrypted job message** (`doc_key`),
which the web app relays unopened — so only the client and the worker ever see
it (same trust model as the message bodies; no second KEM). The worker-facing
blob SAS URLs travel the same way; the worker holds no storage credentials of
its own.

## Key trust

The service's X-Wing public key can be established two ways:

1. **Pinned key** — the client is configured with the key itself
   (out-of-band). A different announced key is irrelevant; a wrong pin fails
   safe (the worker cannot decrypt, nothing readable is ever exposed).
2. **Verified signed announcement** (supports key rotation) — the Ping
   response carries `SignedServicePublicKey`, a JSON announcement of the
   X-Wing key signed with a **hybrid of Ed25519 and ML-DSA-65** (FIPS 204)
   over the payload
   `UTF8("iptr-key:v1:" + public_key + ":" + not_before + ":" + not_after)`.
   **Both** signatures must verify and the current time must fall inside the
   validity window; see
   [`SignedKeyAnnouncement`](../src/IPTranslator.Client/Messaging/SignedKeyAnnouncement.cs).
   The 1984-byte verification key (Ed25519 pk ‖ ML-DSA-65 pk) is the long-term
   trust anchor; the matching signing key is kept offline by the service
   operators and never reaches the web app, so the web app can announce the
   current key but cannot forge one. The production verification key ships in
   [`E2EDefaults`](../src/IPTranslator.Client/E2E/E2EDefaults.cs).

There is deliberately **no unencrypted fallback** in this client:
`DocumentJobClient` refuses construction without a service public key.

## What the server can and cannot see

Can see (needed for auth, quota, billing, operations): API key identity,
job ids and states, document *name* and custom reference you pass to
`Submit`, source/target languages, consumed units, per-segment usage hashes
(salted with a secret only the worker holds), timings, blob sizes.

Cannot see: the document content, dictionaries, translation memories, the
translated result, or the per-job blob key.
